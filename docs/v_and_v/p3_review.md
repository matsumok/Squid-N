# P3 レビュー報告（仕様 vs 実装の乖離・資料上の報告の正確性・仕様自体の妥当性）

本報告は `specs/P3_最小UIと設計.md` と現状実装（主に `sc-app`・`sc-design-jp`・`sc-edit`・関連 `sc-core/sc-section/sc-solver/sc-element/sc-load`）を照合し、
「仕様通りか」「実装が間違っていないか」「資料上の報告が間違っていないか」「仕様自体がこのライブラリの目的（日本の建築構造計算一貫プログラム）に適するか」を審査した結果である。

## 1. 実装の重大バグ（許容応力度周り）

### 1.1 曲げ応力の単位スケール誤り（1000倍のズレ）★致命的

`crates/sc-design-jp/src/allowable_stress.rs:35`
```rust
let sigma_b = forces.m.abs() * 1000.0 / z_eff;
```

- 仕様 §6.4 は `M=100 kN·m=1e8 N·mm`、`Z=5.3333e6 mm³`、`σ=M/Z=18.75 N/mm²`、`fb=F/1.5=156.67`、`検定比=0.1197` を要求（相対 1e-9 の厳密一致）。
- 実装の `M * 1000` は M が `kN·m` で入ったとき `kN·mm` を作る only。`Z[mm³]` で割ると `[kN/mm²]`（= 1000 `[N/mm²]`）となり、これを `[N/mm²]` である `fb` で割ると **検定比が 1000 倍小さくなる**。
- 正しくは `M[kN·m] * 1e6 / Z[mm³]` で `[N/mm²]` を得るか、`MemberForcesAt` の単位を N·mm に統一して `M / Z` とする必要がある。
- 本質的に `MemberForcesAt`（仕様 §6.1）に**単位規定が無い**ことが根本原因（→ §4.1）。構造解析ライブラリとして致命的。数値はプログラム全体で一貫した単位系（SI あるいは N/mm² 系）で扱わねばならず、設計モジュールに入るところで単位変換を明示すべきである。

### 1.2 テスト期待値がバグに追従している（虚偽報告）★致命的

`allowable_stress.rs:191`（`test_steel_check_bending`）
```rust
let expected_ratio = (100.0 * 1000.0 / 5e6) / (235.0 / 1.5);
```
- これは `0.02 / 156.67 ≈ 1.28e-4` を期待値としている。仕様 §6.4 が要求する `0.1197` と**3 桁以上乖離**。
- テストは「通る」が、DoD「規準例題一致」は未達。`cargo test` の緑が DoD 成立を意味しない典型例。
- V&V レポート `docs/v_and_v/README.md` の索引 #13「許容応力度設計 ✅」は**虚偽報告**。正しくは 🔶 または ❌。

### 1.3 RC 設計で Fc をヤング率から取得★致命的

`allowable_stress.rs:93`
```rust
let fc = mat.young;       // Fc（設計基準強度）をヤング率から取得している
let _fc_allow = allowable_concrete_stress(fc, long_term);   // 計算してすぐ無視
```
- `Fc`（設計基準強度 `[N/mm²]`、24 など）とヤング率 `E`（`205000` 等）は完全に別の量。`Material` 構造体（`sc-core/src/model.rs:122`）に Fc フィールド自体が存在しないことも問題。
- `_fc_allow` を計算してアンダースコアで破棄しているため、**コンクリートの許容応力度が検定比に一切反映されない**。
- RC 検定の結果は鉄筋の許容引張だけで決まっており、コンクリートの圧縮検定が抜け落ちている。これは構造解析一貫プログラムとして成立しない。

### 1.4 RC 鉄筋グレードのハードコード

`allowable_stress.rs:100`
```rust
let rebar_grade = "SD345";   // 材料名を見ずに固定
```
- `mat.name` を参照すべき。SD295 や SD390 など他グレードの断面で全て SD345 として検定される。

### 1.5 SD345 短期許容引張の上限誤り

`allowable_stress.rs:80`
```rust
(base * 1.5).min(295.0)     // SD345 でも上限を 295 にしている
```
- SD345 の F値は 345。短期許容引張の上限は F（=345）。215×1.5=322.5 < 345 なので、本来 322.5 が採用されるべき（あるいは令90条表の 345）。
- 実装は SD295 の F値（295）を SD345 にも適用しており、過小な許容値を出す。
- テスト `test_rebar_allowable_short_upper_bound` が `295.0` を期待しているため、テストは通るが**規準値として誤り**。これも虚偽報告に該当し得る。
- 仕様 §6 は「短期 = 長期 × 1.5（上限あり）」としか書かず、**上限値を明記していない**（→ §4.2）。令90条表では上限=F値である。

### 1.6 鋼種の F値テーブル不備

`allowable_stress.rs:7-12`
- `SS490 => 235.0` は誤りの可能性（SS490 の F値は板厚≤40 で 285）。原典照合（R4/R23）で確定すべき項目だが、暫定値としても SS400 と同値は不適切。
- `SM490` が未収載（SN490 のみ 325）。JIS 鋼種のカバレッジが実用範囲で不足。

### 2.7 GUI コードが CI/ローカルで一切ビルド・検証されていない★重大
- `egui 0.34.3` が rustc 1.92 を要求するのに対し、現環境は rustc 1.89.0。
  `cargo build -p sc-app --features gui` は依存解決で即座に失敗し、GUI コードは1行もコンパイルされていない。
- このため §2.2 の `StaticResult`/`StaticOnce` 型不一致（`app.rs:19` が `StaticResult`、実装は `StaticOnce`）を含め、
  `app.rs`・`tables/*`・`design_view.rs`・`viewer/mod.rs` の全 GUI 実装がコンパイラ・clippy・テストの何れにもかかっていない。
- CI が `--all-targets` で回っていても `gui` feature が既定で OFF（`sc-app/Cargo.toml` `default = []`）なら同様。
  P3 DoD の【目視】項目だけでなく【自動】項目（Undo 往復など）も gui feature 配下に置かれていないか確認が必要。
- 仕様 P0 §2.4 の「`gui` は opt-in」方針自体は妥当だが、P3 の成果物が一切検証されない状態で「P3 完了」と報告されているのは問題。

### 2.8 編集トランザクションの二重実装
- `sc-app/src/command.rs` と `sc-edit/src/lib.rs` に**同一の `EditCommand`/`UndoStack` が重複実装**。
- `sc-app` は `sc-edit` に依存していない（`sc-app/Cargo.toml` 参照）。
- 仕様 §3.3 の意図（MCP と共通化の土台＝単一ライタ）からすると `sc-edit` に統一すべき。`sc-app/command.rs` は古い位置のまま残った dead code。
- `sc-edit::SetNodeCoord`（`lib.rs:94`）はインデクスチェックが無く、存在しない NodeId を渡すと panic する（`sc-app` 側はチェック有り）。統合時に弱い方を採用しないよう注意。

## 4. 仕様書自体の問題点（このライブラリの目的への適合性）

### 4.1 MemberForcesAt に単位規定が無い★根本
- 仕様 §6.1 の `MemberForcesAt { pos, n, q, m }` に単位が書かれていない。M が `kN·m` か `N·mm` か `N·m` か不明。
- 構造解析ライブラリでは解析系（P1/P2）と設計系（P3 §6）の間で単位を固定するか明示的に変換するのが鉄則。本仕様はこれをサボった結果、実装が §1.1 の 1000 倍バグを生んだ。
- **是正案**: `MemberForcesAt` の doc に「`n: N`, `q: N`, `m: N·mm`, `pos: m（部材軸方向の無次元位置 0..1 も可）」」を明記。P2 側 `MemberForces` と単位を合わせる。

### 4.2 許容応力度の「上限値」が未記載
- §6「短期 = 長期 × 1.5（上限あり）」の「上限」が未記載。令90条表では上限=F値（SD295=295, SD345=345, SD390=390）。
- 「AIJ 私的規準だけ外部データ」と仕様は分けるが、**F値テーブルと短期上限は令由来（公有）**であり本書で確定できるはず。外部データに後回しすると実装が勝手に誤値を埋める（§1.5）。
- **是正案**: §6 に SD295/SD345/SD390 の長期・短期表と、短期 `min(長期×1.5, F)` の式を明記。

### 4.3 RC 一次設計の検定式が σ=M/Z しか無い
- §6.4 の「検算例」は鋼梁のみ。RC の曲げ・せん断の検定式が一切示されていない。
- 構造一貫プログラムとして RC は必須（日本の建築の大部分は RC か SRC）。P3 の完了基準「一次設計が一通り通る」を RC について達成するには、引張鉄筋比 `a_t/(b·d)`、`M/(b·d²)` から `ft` 比を取る経路、せん断検定 `Q/(b·j)` と `αs·fw` 等の式が要る。
- 現状の「σ=M/Z を ft で割る」は RC 検定としては物理的意味を持たず、構造技術者が出力を信頼できない。
- **是正案**: §6 に RC 曲げ（長期・短期）・せん断の基本式を追加。SRC は「骨格のみ」でよいが、RC の基本式は P3 の責任範囲。

### 4.4 編集トランザクションの所属が未確定
- §3.3 は `sc-app/src/command.rs` に置くと書くが、§0.3/§9.2 は「MCP（P8）と共通化」と述べる。共通化するなら `sc-edit` クレートが自然。
- 実際に `sc-edit` が作られているが、仕様書がそれを追認していない。ジュニア実装者が双方に実装してしまう（現状）。
- **是正案**: §3.3 を「`sc-edit` クレートに `EditCommand`/`UndoStack` を置き、`sc-app`・MCP はそれを参照」と改め、`sc-app/src/command.rs` は削除。

### 4.5 ResultsBundle の型名と P2 実装の不一致
- 仕様 §2 の `ResultsBundle.statics` は `StaticResult`。P2 実装は `StaticOnce`。仕様が P2 の確定 API 名とずれている。
- 仕様書群は「ライブラリAPIは調査不要・写譯で埋める」と謳う（README §5）以上、型名は実装と一致している必要がある。

## 5. 推奨対応（優先度順）

1. **【最優先・自動テストで即検知可能】許容応力度の単位と期待値を修正**
   - `MemberForcesAt` に単位 doc を付与（N・N・N·mm）。
   - `allowable_stress.rs` の `* 1000.0` を削り、`M[N·mm] / Z[mm³]` で `[N/mm²]` を得る。
   - テスト `test_steel_check_bending` を仕様 §6.4 の `M=1e8 N·mm, Z=5.3333e6, σ=18.75, 検定比=0.1197` に一致するよう書き直す。
2. **RC 検定の全面改修**
   - `Material` に `Fc`（コンクリート強度）と鉄筋グレードを表現するフィールドを追加（P4 と連動）。
   - 鉄筋グレードを `mat.name` から取得。SD345 短期上限を F=345 に。
   - RC 曲げ・せん断の基本式を実装し、コンクリート許容応力度を検定比に反映。
3. **V&V 索引 #13 の修正**
   - `docs/v_and_v/README.md` の #13 を ❌ に変更。フェーズを P3 に修正。§6.4 数値例のテストを追加で合格後に ✅ に戻す。
4. **編集トランザクションの統一**
   - `sc-app/src/command.rs` を削除し `sc-edit` に統一。`sc-app` は `sc-edit` に依存。
   - `sc-edit::SetNodeCoord` にインデクスチェックを追加。
5. **DoD 未達項目の実装**
   - T3 解析起動メソッド、T4/T5 ビューア・応力図、T7 checks 生成ドライバを順次実装。
   - 並行して `pending_items.md` に P3 の実状況を過小申告なく記載。
6. **GUI feature の CI 検証環境を整備**
   - rustc 1.92（または egui 0.34 が要求する版）を CI/ローカルに整備し、`cargo build/clippy/test -p sc-app --features gui --all-targets` が通るようにする。
   - これが無いと §2.2/§2.7 の型不一致や GUI 側の回帰を検知できない。
7. **仕様書の改訂**
   - §6 に SD 鉄筋の長期・短期表と上限式を明記。
   - §6.1 `MemberForcesAt` に単位を明記。
   - §6 に RC 曲げ・せん断の基本式を追加。
   - §3.3 のコマンド所属を `sc-edit` に変更。
   - §2 `ResultsBundle.statics` の型名を `StaticOnce` に修正。

## 6. 結論

P3 は「スケルトンはあるが、一次設計の数値が正しく出ない」状態である。とくに許容応力度は仕様 §6 が「厳密・自動テストで確定できる」と明記する部分であり、ここが 1000 倍のズレを抱えたまま `cargo test` が緑で V&V 索引も ✅ になっていることは、本ライブラリが「構造技術者が信頼して使える一貫プログラム」を目指す上で**最も警戒すべき品質リスク**である。実装・テスト・資料報告の三層で整合を取り直す必要がある。
