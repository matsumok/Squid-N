## 2. T0: sc-mcp 雛形（rmcp）

設計書 §13。**rmcp 1.7 の実 API（確認済み・そのまま使える）。** 調べ直す不要。

確定しているパターン（rmcp 1.7、`features = ["server","macros","transport-io"]`）:
- ツール定義：`impl` ブロックに `#[tool_router]`、各メソッドに `#[tool(description="…")]`。
  メソッドは `async fn`、引数は `Parameters(args): Parameters<引数型>`、戻り値は
  `Result<CallToolResult, rmcp::ErrorData>`。
- サーバ実装：`#[tool_handler] impl ServerHandler for ...`（`get_info` で `ServerInfo` を返す）。
- 起動：`use rmcp::ServiceExt;` の `.serve(stdio()).await?` → 返った service を `.waiting().await?`。
  `stdio()` は `rmcp::transport::stdio`。HTTP は `transport` の streamable-http（feature 追加）。

```rust
// sc-mcp/src/lib.rs
use rmcp::{tool, tool_router, tool_handler, ServerHandler, ServiceExt, ErrorData};
use rmcp::handler::server::tool::{Parameters, ToolRouter};
use rmcp::handler::server::router::tool::ToolRouter as _; // 生成フィールド型（版により再エクスポート位置差異あり）
use rmcp::model::{CallToolResult, Content, ServerInfo};
use rmcp::transport::stdio;
use std::sync::Arc;
use tokio::sync::Mutex;

#[derive(Clone)]
pub struct StructCalcServer {
    /// 単一ライタ：モデルは1つの所有者が直列に編集（R29）。Mutex で直列化。
    state: Arc<Mutex<ServerState>>,
    tool_router: ToolRouter<Self>,   // #[tool_router] が要求する生成フィールド
}
pub struct ServerState {
    pub model: sc_core::model::Model,
    pub undo: sc_edit::UndoStack,          // 編集コマンド（P3 §3.3）。下記 layering 注意
    pub jobs: JobRegistry,                 // §5
    pub results: Box<dyn sc_io::results::ResultStore>, // P2 §6
}

#[tool_router]
impl StructCalcServer {
    pub fn new(state: ServerState) -> Self {
        Self { state: Arc::new(Mutex::new(state)), tool_router: Self::tool_router() }
    }
    // 各ツールは §3/§4/§5/§6 で #[tool] を付けて定義する。
}

#[tool_handler]
impl ServerHandler for StructCalcServer {
    fn get_info(&self) -> ServerInfo { ServerInfo::default() } // 名称・能力を設定
}

// 起動（main 側、#[tokio::main]）:
//   let service = StructCalcServer::new(init).serve(stdio()).await?;
//   service.waiting().await?;
```

> **import 行の微差について:** 関数名・属性（`#[tool]`/`#[tool_router]`/`#[tool_handler]`・
> `Parameters`・`CallToolResult`・`serve`/`stdio`）は 1.7 で確定。`ToolRouter` 等の**再エクスポート
> パスだけ**版で前後し得るので、コンパイラの指示に従って use を直す（調査ではなく機械作業）。

> **⚠ layering 修正（P3 からの調整・必須）:** P3 では `EditCommand`/`UndoStack` を `sc-app` に置いたが、
> `sc-mcp` と `sc-app` は**同じ操作層のピア**であり、`sc-mcp → sc-app` 依存は `xtask check-deps`
> （P0 §2.5）違反になる。よって **`EditCommand`/`UndoStack` を共有下位クレート（`sc-core` 直上の
> `sc-edit`、あるいは `sc-core` 内）へ移し**、`sc-app`・`sc-mcp` の双方がそこに依存する形にする。
> P3 のコードはこの移設に合わせて参照先を変える（GUI と MCP が同一の編集経路＝単一ライタを共有）。

**DoD（T0）:** stdio で rmcp サーバが起動し、MCP Inspector から `tools/list` が取得できる。

## 3. T1: 読取系ツール

設計書 §13。`schemars` で入出力スキーマを型安全に生成。

```rust
// sc-mcp/src/tools_read.rs（§2 と同じ impl ブロック内に #[tool] で追加）
// model.load / model.query / result.get

#[derive(schemars::JsonSchema, serde::Deserialize)]
pub struct QueryArgs { pub kind: String /* node/member/section */, pub filter: Option<String> }
#[derive(schemars::JsonSchema, serde::Serialize)]
pub struct QueryResult { pub items: Vec<serde_json::Value> }

#[tool_router]   // §2 の new() と同じ impl にまとめる（ここでは抜粋）
impl StructCalcServer {
    #[tool(description = "節点・部材・断面を検索する")]
    async fn model_query(&self, Parameters(args): Parameters<QueryArgs>)
        -> Result<CallToolResult, ErrorData>
    {
        let st = self.state.lock().await;                 // 読み取りロック
        let result = /* st.model から args.kind/filter で収集 */ QueryResult { items: vec![] };
        // 確定 API: 構造化結果は Content::json、テキストは Content::text
        Ok(CallToolResult::success(vec![Content::json(result)?]))
    }
}
```

> `#[tool]` を付けたメソッドは自動でクライアントに公開され、引数型（`schemars::JsonSchema` 派生）から
> 入力スキーマが生成・提示される（ジュニアがスキーマを手書きする必要はない）。

**DoD（T1）:** `model.load`→`model.query`→`result.get` がスキーマ提示つきで動作。

## 4. T2: 書込系ツール（単一ライタ・破壊的操作の識別）

設計書 §13 / 付録A R29。

- **編集は P3 §3.3 の `EditCommand` を生成して `UndoStack::run` に流す**（GUI と同一経路＝単一ライタ）。
- **書込系（編集・保存・上書き）は破壊的操作を識別子付きで返し**、クライアントに確認させる前提の
  レスポンスにする。

```rust
// sc-mcp/src/tools_write.rs
// model.edit : 部材追加・断面変更・荷重設定 → EditCommand 化して適用（Undo 可能）
// model.save : 上書き保存（.scz）。破壊的なので操作IDを返す

#[derive(schemars::JsonSchema, serde::Serialize)]
pub struct WriteResult {
    pub op_id: String,        // 破壊的操作の識別子（クライアント確認・Undo 参照用）
    pub undoable: bool,
    pub summary: String,      // 何が変わるか（確認用）
}
```

> **設計意図:** GUI（P3）・MCP（P8）の編集が**同じ `EditCommand`＋単一 `UndoStack`** を通ることで、
> モデルの一貫性（単一ライタ）が保たれ、Undo/Redo・直列化が共通になる。

**DoD（T2）:** MCP からの編集が Undo 可能。書込系が `op_id` を返す。GUI と MCP が競合せず直列に適用。

## 5. T3: 非同期ジョブ（解析の投入＋進捗）

設計書 §13。**解析は時間がかかるため、ジョブ投入＋進捗（notification/resource）の非同期パターン**。

```rust
// sc-mcp/src/jobs.rs
pub type JobId = String;
#[derive(Clone, serde::Serialize, schemars::JsonSchema)]
pub enum JobStatus { Queued, Running { progress: f32 }, Done { result_ref: String }, Failed { error: String } }

pub struct Job { pub id: JobId, pub kind: JobKind, pub status: JobStatus }
pub enum JobKind { LinearStatic, Eigen, Pushover, TimeHistory, DesignCheck }
pub struct JobRegistry { /* JobId -> Job。進捗更新で notification 送出 */ }

// ツール:
//   analysis.run   : 解析種別＋パラメータを受け、Job を作って **即 JobId を返す**（非同期）。
//                    実計算は tokio タスクで実行（P2/P5/P6）。進捗は notification/resource で通知。
//   analysis.status: JobId の状態取得（ポーリング用）
//   result.get     : Done の result_ref から結果取得（P2 §6 の部分読出し）
```

- 実計算は**別タスク**で走らせ、サーバの応答性を保つ（編集の単一ライタとは分離。読み取り中心）。
- 進捗は **MCP notification**（または resource 更新）でクライアントへ。長時間はチェックポイント（P6 §5）と連携可。
- 決定性：解析本体は単一スレッド設定でビット一致（P0〜P6 を踏襲）。ジョブ管理の並行性は結果値に影響させない。

**DoD（T3）:** `analysis.run` が即 `JobId` を返し、進捗通知後に `result.get` で結果取得。
線形静的・固有値・プッシュオーバー・時刻歴をジョブとして実行できる。

## 9. 補足（ジュニア向け）

### 9.1 MCP とは（→ §2）
MCP（Model Context Protocol）は、AI エージェント等のクライアントが「ツール（関数）」「リソース」を
標準プロトコルで呼ぶ仕組み。rmcp は公式 Rust SDK。`#[tool]` を付けた関数が外部に公開され、入出力
スキーマ（schemars）がクライアントに提示される。トランスポートは stdio（ローカル）／HTTP（リモート）。

### 9.2 なぜ非同期ジョブか（→ §5）
解析は数秒〜数分かかる。同期ツールで待たせるとクライアントがブロックし、進捗も分からない。そこで
`analysis.run` は**即ジョブIDを返し**、実計算は裏で走らせ、進捗を notification で送る。クライアントは
完了通知後に結果を取りに来る。長時間ジョブはチェックポイント（P6）で中断・再開できる。

### 9.3 単一ライタ（→ §4）
モデルを書き換える経路（GUI・MCP）が複数あると競合する。本システムは**編集を1つの `EditCommand`＋
`UndoStack` に集約**し、直列に適用する（単一ライタ。R29）。読み取り（query/result）は並行してよい。

## 10. 用語（P8 で増えるもの）
| 用語 | 意味 |
|---|---|
| MCP | Model Context Protocol。ツール／リソースの標準プロトコル |
| rmcp | MCP の公式 Rust SDK（1.x） |
| ツール | MCP で公開する関数（model.edit, analysis.run 等） |

| 単一ライタ | モデル編集を1経路に直列化する方針（R29） |
| schemars | Rust 型から JSON スキーマを生成（型安全な提示） |
| ST-Bridge | 日本の構造分野のデータ交換標準（XML） |
| 意味的往復 | バイト一致でなく対象範囲の意味が保存される往復 |
