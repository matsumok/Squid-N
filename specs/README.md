# 実装仕様書（フェーズ別）

このディレクトリは、ルートの『建築構造計算一貫プログラム 実装設計書（開発指示書）』
（`構造計算一貫プログラム_実装設計書.md`）をもとに、**フェーズ単位**で
「ジュニアエンジニアがそのまま着手できる」粒度に落とした実装仕様書を置く場所です。

- 親文書（設計書）= **何を作るか・なぜそう作るか**（方針・確定事項）
- 本ディレクトリ（仕様書）= **どう作るか**（ファイル構成・型シグネチャ・実装手順・テスト・DoD）

仕様書中で `設計書 §4.2` のように参照している番号は、すべて親設計書の章番号です。

## 仕様書の読み方（ジュニア向け）

1. 各仕様書の「0. このフェーズについて」でスコープ（含む／含まない）を必ず確認する。
2. 「1. タスク一覧と依存順序」の順に着手する。前のタスクが終わらないと次に進めない。
3. 各タスクの **実装ステップ** を上から実施し、**DoD（受け入れ条件）** を満たしたら完了。
4. コード断片は「雛形」。シグネチャ（型・関数名・引数）はそのまま使い、中身を実装する。
5. **ライブラリAPIは調査不要**：主要外部ライブラリ（faer 0.24・rmcp 1.7・arrow/parquet 59・
   egui/eframe 0.34）は**実APIを確認して具体コードに落とし込み済み**で、バージョンは
   [P0 §2.2](P0_基盤.md) の `[workspace.dependencies]` に一元固定。写経して埋めればよい
   （`import` 行の再エクスポート位置の微差だけはコンパイラの指示に従う＝調査ではなく機械作業）。
5.5 **法令・規準の埋込値は [原典照合リスト.md](原典照合リスト.md) で技術リードが着手前に1回だけ
   原典照合（サインオフ）する。** 実装者はこのリストを使わない（実装者は調べない）。値を埋め込んだことで
   「誰も照合しない」状態になるのを防ぐ安全網。最優先は Ds（βu/壁量変動）・許容応力度（SD345の径区分）。
6. **残る「要検証／外部データ」は2種類だけ**で、いずれも*ライブラリAPI調査*ではない：
   - **力学の閉形式**（ティモシェンコ12×12・Newmark更新式・Ai式・CMQ・板たわみ係数 等）は
     本文に**式を明示**済み。出典は併記（実装後に各DoDの数値で照合）。
   - **規準・告示の係数値**（Z/Rt/Tc・Ds値表・許容応力度・履歴則係数 等）は設計方針により
     **外部データファイル化**（版管理）。これは出典告示からの**データ入力作業**であって、
     ジュニアのスキルに依存するAPI調査ではない。出典（告示番号・規準ページ）は本文に明記。
