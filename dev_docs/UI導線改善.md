# UI 導線改善（実装済み機能と UI の接続）

作成日: 2026-07-05
対象ブランチ: `claude/ui-navigation-flow-dgwe2j`
関連文書: `docs/申し送り.md`（残課題 3・4 に対応）、`specs/UI設計.md`（§1.1 双方向連動・§2 反復動線）

## 1. 調査で確認したギャップ

GUI (squid-n-app) から到達できない実装済み機能、および導線が弱い箇所を全数調査した結果:

### 実装済みだが入口がゼロだった機能
| 機能 | 実装場所 | 状態 |
|---|---|---|
| ST-Bridge 2.0 入出力 | `squid-n-io::stbridge` (export/import とも実装済) | メニューに項目なし・参照 0 件 |
| 荷重組合せ | `squid-n-load::combo::auto_combinations` + `squid-n-solver::Analysis::linear_combination` + `Model.combinations` | 作成・実行 UI なし・参照 0 件 |
| 保有水平耐力判定・Ds | `squid-n-design-jp::holding_capacity::{check_holding_capacity, ds_value, qud_by_story}` | 設計タブ未統合・参照 0 件 |

### 導線が弱かった箇所
- 変形図が常に `statics.first()` を表示（ケースを切り替えても変形図が変わらない）。
- ナビゲータの「結果ケース」がラベル表示のみでクリック不可。
- 断面テーブルの行が選択不可（節点・部材・荷重ケースは選択→連動があるのに断面だけ非対応）。
- 設計タブの NG 部材から断面編集への遷移がない（UI設計 §2 の「反復動線」未達）。
- ナビゲータ「階/レベル」が「（未実装）」プレースホルダのまま（階自動生成は実装済みなのに）。
- ナビゲータ「部材グループ」がクリックしても何も絞り込まない偽フィルタ。
- 解析実行後の結果タブ自動遷移がプッシュオーバー・時刻歴のみ（線形静的・地震静的は遷移しない不統一）。

## 2. 導線設計と実装内容

### 2.1 ファイルメニュー → ST-Bridge 入出力
「📥 ST-Bridge 読込…」「📤 ST-Bridge 書出…」をファイルメニューに追加。
読込は validate 後にモデル差し替え（プロジェクトパスは None のまま＝SCZ とは別系統）。
ヘッドレステスト可能な `import_stbridge_from` / `export_stbridge_to` をロジック層に置き、
rfd ダイアログは gui feature 側の薄いラッパとした（既存の scz 保存/読込と同じ構造）。

### 2.2 荷重タブ → 荷重組合せの作成、解析タブ → 組合せ実行
- `squid-n-edit` に `AddCombination` / `DeleteCombination` を新設（undo 可、MCP と共通経路）。
- 荷重タブに「荷重組合せ」セクション: 一覧（名前・項の内訳）・削除・
  DL/LL/地震X/地震Y/積雪のケース割当てから `auto_combinations` で一括生成。
- 解析タブに「荷重組合せ」グループ: 組合せを選んで実行 → `linear_combination`。
- 結果は `ResultsBundle.combos: Vec<(String, StaticOnce)>` に名前付きで保持し、
  実行後は検定 (`run_design_check`) も更新する。

### 2.3 結果ケースの選択導線（変形図の表示対象）
- `App::current_static()` を導入: ナビゲータで選択した結果（静的ケース or 組合せ）→
  未選択なら最後に実行した結果、の順で表示対象を解決する。
- ナビゲータ「結果ケース」をクリック可能にし、変形図・層指標がクリックで切り替わる。
- 変形図の `statics.first()` 固定バグを `current_static()` に置換。

### 2.4 設計タブ → 保有水平耐力（ルート3）セクション
- 前提データの充足状態をガイド表示（階未定義→自動生成へ、プッシュオーバー未実行→解析タブへ、
  のジャンプボタン付き）。
- 架構種別（RC/S × ラーメン/壁・ブレース）と部材ランク（FA〜FD）を選択 → `ds_value` で Ds を決定
  （部材ランクの自動判定は未実装のため選択値による簡易運用であることを UI に明記）。
- Qud は `qud_by_story`（地震重量・Z・Rt・T1）、Fes/Rs/Re は既存の層指標計算を利用し、
  `check_holding_capacity` で Qu ≥ Qun = Ds·Fes·Qud を層別判定・色分け表示。
- あわせて squid-n-solver のプッシュオーバーが `CapacityPoint.story_shear`/`story_drift` を
  常に空 Vec で返していた欠落（Qu が常に 0 と判定される）を発見し、層せん断（層以上の
  水平内力の上層からの累積）と層間変位（剛床マスター変位差）の算定を実装した。

### 2.5 細かい導線の統一
- 断面テーブルの ID 列をクリック可能に（`focus_section` 連動）。断面選択時はインスペクタに
  断面詳細＋使用部材の 3D ハイライトボタンを表示。
- 設計タブ検定表に「断面」列を追加し、クリックでモデルタブ断面サブタブへジャンプ
  （NG 部材 → 断面編集の反復動線）。
- ナビゲータ「階/レベル」を実データ表示に置換（空なら解析タブへの誘導ボタン）。
- ナビゲータ「部材グループ」クリックで該当部材群を 3D ハイライト（偽フィルタ解消）。
- 線形静的・地震静的も成功時に結果タブへ自動遷移（プッシュオーバー・時刻歴と統一）。

## 3. 第2ラウンド（前節の見送り項目の実装）

第1ラウンドで見送った項目のうち、以下を実装した。

### 3.1 スラブ入力 UI + CMQ データ供給（旧・見送り1）
- squid-n-edit に `AddSlab`（id はコマンド内採番）／`DeleteSlab`（後続 id 繰り上げ、undo 可）を新設。
- モデルタブに「スラブ」サブタブ: 境界節点4つ・床荷重（UI は kN/m²、内部 N/mm²）・
  分配法（三角/台形 45°法・一方向・負担面積）の入力と一覧・削除。
- `App::refresh_beam_loads()`: `distribute_slab` が返す**辺インデックス**（floor.rs の仕様）を
  境界節点対→実部材 ID に対応付けて `beam_loads` を供給。CMQ 表示モード中は毎フレーム再計算し
  モデル編集へ常に追従する。これで CMQ 図が初めて実データで描画される。

### 3.2 時刻歴の拡張配線（旧・見送り3）
- 方向選択（X/Y）: サンプル波・CSV 波形とも `accel_y` へ配線。
- 減衰モデル選択: 剛性比例（従来）／Rayleigh（h1・h2、`eigen(2)` の ω1・ω2 から算定）。
- 積分法選択: Newmark-β（従来）／HHT-α（α=−0.1、`Analysis::time_history_hht` を新設して配線）。
- ソルバー側の代表応答記録（`ResponseHistory`）が X 方向ハードコードだったため、
  **入力加速度の絶対値和が大きい方向を自動選択**する方式に一般化
  （`node_disp_x`→`node_disp` 等に改名、`record_dir_y` フラグ追加）。

### 3.3 既存断面のパラメトリック再編集（旧・見送り4）
- 断面エディタに「✏ 選択断面へ適用」: フォーム内容で選択断面を `EditSectionShape` により再定義
  （影響部材数をホバー表示、undo 可、名称維持）。

### 3.4 階の地震重量の手動編集（旧・見送り6）
- squid-n-edit に `SetStoryWeight` を新設し、解析タブ「階の定義」で W[kN] を
  DragValue 編集（ドラッグ終了時に 1 コマンド発行、undo 可）。

### 3.5 解析の非同期化（旧・見送り2、P8）
- プッシュオーバー・時刻歴を計算（純関数）／適用／同期実行の3層に分離し、
  バックグラウンドスレッド＋mpsc のジョブ基盤（`AnalysisJob`）で実行。
- 実行中は解析ボタンを無効化し、ステータスバーに経過表示。完了時に結果タブへ遷移。
- 線形静的・固有値・地震静的・組合せは軽量なため同期実行のまま。

## 4. 第3ラウンド（前節の残課題の実装）

### 4.1 固有値解析（部分空間反復）の頑健化（旧・課題3）
根本原因は2つの複合だった: (a) 射影質量行列のランク落ち時に Cholesky 失敗→対角近似へ
フォールバックし **f64::MAX を結果に注入**していた、(b) 開始ベクトルが自由度番号順で
質量分布を無視していた。修正として、射影質量行列を固有分解して質量を持つ部分空間のみで
解き（質量ゼロ方向は +∞ を割当てて明示エラー化）、開始ベクトルを Bathe の定石
（質量比例＋k_ii/m_ii 昇順）に変更、部分空間サイズも q=min(2p, p+8) に是正。
portal_frame の eigen(2) が正常値（T1=0.232s, T2=0.221s）を返すようになった。
質量ランク不足時は「モード数を減らすか質量を追加」の日本語診断を返す。

### 4.2 PushoverStep の構築（旧・課題4）
荷重制御（λ）・変位制御（λ=1 到達後のため 1.0 固定、根拠コメント付き）・弧長法
（arc_lambda）の3経路すべてで `steps` を capacity_curve と同数構築する。

### 4.3 SectionShape の core 移設と Section の形状保持（部材ランク自動判定の前提）
仕様（UI設計 §4.2「Section は SectionShape の派生」）どおり `SectionShape` 一族を
squid-n-core::section_shape へ移設（squid-n-section::shape は再エクスポートで互換維持）し、
`Section.shape: Option<SectionShape>`（serde default で .scz 後方互換）を追加。
`to_section` 経由の断面は形状を保持し、カタログ断面も名前（"H-400x200x9x12x13" 等）から
H/角形鋼管/丸鋼管の形状をベストエフォート復元する（数値プロパティはカタログ表値のまま）。

### 4.4 鋼部材ランクの幅厚比自動判定（旧・課題1の鋼側）
`max_width_thickness(shape)` を design-jp に新設（H: max(b/2tf, ウェブ内法/tw) 等、
式の根拠コメント付き・境界値は既存 RankCriteria 同様に要原典照合）。
設計タブの保有水平耐力セクションに「自動判定(鋼・幅厚比)」を追加: 層ごとに所属鋼部材
（所属階は節点 story の最大＝柱は上端の階）の最悪ランクから Ds を層別に決定し、
採用ランク列を表示。RC・形状未設定部材は選択値フォールバック。

### 4.5 時刻歴の X+Y 同時入力（旧・課題5）
方向選択に「X+Y」を追加（同一波形を両方向へ同時入力する簡易仕様）。
CSV は2列（1列目X・2列目Y、gal）として読み、列不足は明示エラー。

## 5. 第4ラウンド（前節の申し送りの実装）

### 5.1 RC 部材ランクの自動判定（旧・申し送り1）
squid-n-design-jp に `rc_capacity` モジュールを新設し、閉形式の略算で Qsu/Qmu を算定:
- Mu = 0.9·at·σy·j（j=7d/8、skeleton のテストで照合済みの簡易式と同一）
- Qmu = 2Mu/h0（反曲点中央仮定）
- Qsu = {0.068·pt^0.23·(Fc+18)/(M/(Q·d)+0.12) + 0.85·√(pw·σwy)}·b·j
  （荒川mean式系。せん断スパン比 1〜3・pw≤0.012 にクランプ、軸力項は安全側に省略）
設計タブの自動判定が RC 矩形断面（`RcRect` の配筋・Fc 付き材料）にも適用され、
`rc_member_rank` によるせん断余裕度判定が層 Ds に反映される。RC 円形・形状未設定・
Fc 未設定は選択値フォールバック。係数はすべて「要・原典照合」コメント付き。

### 5.2 埋め込み値の原典照合リスト整備（旧・申し送り2）
specs/原典照合リスト.md に部材ランク自動判定の全埋め込み値（幅厚比境界・算定式、
Qsu/Qmu 境界、荒川式係数、σy/σwy 既定値）の照合行を追加。**照合そのもの
（専門家サインオフ）は未実施**であり、実数の妥当性は原典照合を要する。

### 5.3 squid-n-gpu のビルド不整合の修正（旧・申し送り5）
原因は cubecl 依存の `default-features = false` により `std`/`channel-mpsc` が
落ちること（+上流 cubecl-runtime 0.3 の feature ゲート漏れ）。feature 指定の是正で
`--all-features` ビルド・clippy とも squid-n-gpu 起因のエラーは解消。

### 5.4 その他
- 時刻歴 X+Y の CSV は2列（X列・Y列）の独立波形として読み込み済みだった
  （前節の記述を訂正。同一波形の複製になるのは生成サンプル波のみ）。
- 階高・所属節点の手動編集は前節どおり意図的に見送り（節点編集→再生成が正しい UX）。

## 6. 第5ラウンド（原典照合以外の申し送りの実装）

### 6.1 S 幅厚比の F 値スケーリング（旧・申し送り2）
`s_member_rank_scaled` を新設し、境界値（F=235 基準）に √(235/F) を乗じて判定。
F 値は材料名の前方一致（最長一致）で引く `steel_f_value_prefix` を追加
（既存の許容応力度計算の完全一致経路は挙動不変）。

### 6.2 RC ランクの精緻化（旧・申し送り4の一部）
- Qsu に軸力項 0.1·σ0·b·j を追加（σ0 は 0〜0.4Fc にクランプ、引張は 0。要・原典照合）。
  σ0 は最後に実行した静的解析の部材軸力から算定する簡易運用
  （始端断面力の符号規約は既存テストで圧縮正を確認済み）。
- 内法スパンは幾何長から両端剛域（`rigid_zone`）を差し引く。
- skeleton のファイバ解析 Mu との突合は引き続き将来課題。

### 6.3 squid-n-mcp の修復（旧・申し送り3）
原因は2層: (a) コードが rmcp 1.x の実在しない API パスを前提にしていた
（設計書 P8 の API 例自体が誤り）、(b) schemars 0.8/1.x のメジャー版不一致、
(c) `EditCommand`/`ResultStore` トレイトに `Send` 境界がなく
`rmcp::ServerHandler: Send+Sync` を満たせない。(a)(b) は rmcp 1.7 の実 API へ
追従+mcp のみ schemars 1 系に変更、(c) は両トレイトへ `Send` を追加して解消
（コマンド/ストアはプレーンデータで全実装が自然に満たす）。
これで **`cargo clippy --workspace --all-targets --all-features -- -D warnings` が
ワークスペース全体で初めてクリーン**になった。

## 7. 第6ラウンド（前節の申し送りの実装）

### 7.1 MCP `analysis_run` のジョブ実処理（旧・申し送り3）
ジョブ登録のみだった `analysis_run` を実ジョブ化: 登録→`Running` 遷移→モデル複製→
`spawn_blocking` で解析（線形静的・先頭ケース）→完了時に `Done{result_ref}`／
`Failed{error}` へ更新（panic も `JoinError` 経由で Failed に変換）。ツール応答は
job_id を即時返しブロックしない。`analysis_status` が結果 JSON を返せることを
tokio テスト2件で確認。LinearStatic 以外の JobKind・結果ストア連携は将来課題。

### 7.2 specs/P8 の rmcp API 例の訂正（旧・申し送り2）
「確認済み」と記載されていた実在しない API パス（ToolRouter/Parameters のパス、
`ServerInfo` 構築、`CallToolResult::error`）を rmcp 1.7 の実 API と現実装に合わせて
訂正し、訂正日と実装状況の注記を追加。

### 7.3 RC ランクの精緻化・続き（旧・申し送り4）
- **ファイバ解析 Mu との突合テスト**を追加: 代表断面（b=400×D=600・at=1935・Fc=24）で
  簡易式 Mu／ファイバ Mu ≈ **0.850**（許容域 0.7〜1.3、skeleton 既存テストの
  30% 規律に整合）。略算の妥当域を回帰で担保。
- **Qsu 軸力の長期優先**: σ0 の取得元を「先頭荷重ケース（長期相当）の静的結果を優先、
  なければ最後の静的結果」に変更（LoadCaseId(0) が地震静的の結果格納と重複し得る
  既存仕様の制約はコメントに明記）。

## 8. 第7ラウンド（結果キーの二重使用解消）

### 8.1 LoadCaseId(0) の結果キー二重使用の解消（旧・申し送り3）
静的結果の格納キーを `StaticCaseKey { User(LoadCaseId), Seismic(SeismicDir) }` に分離し、
地震静的(Ai)の結果がユーザーケース 0 と同居する既存仕様（docs/申し送り.md §4）を解消。
- `ResultsBundle.statics` のキーを型で区別（`StaticKey::Case` はネスト方式を採用。
  current_static・ナビゲータが statics キーとの等値比較1本で書けるため）。
- 副次効果: **地震静的の X/Y 方向の結果が共存**できるようになり、ナビゲータに
  「地震静的 (X方向)/(Y方向)」として個別に選択可能。σ0 の長期軸力判定に付いていた
  「先頭ケース id=0 だと地震結果と区別できない」制約も型レベルで解消。
- レポート CSV・節点荷重編集対象のフォールバック・関連テストを追従。
  「ユーザーケース0と地震結果が衝突しない」ことを直接検証するテストを追加。

## 9. 第8ラウンド（MCP ジョブの拡充）

### 9.1 結果ストアの具象実装（squid-n-io）
`FsResultStore::open(dir)` を新設: ディレクトリ配下に Parquet ファイル+manifest.json を
置く `ResultStore` の初の具象実装。ライタ finish 時の保留エントリを `sync()` で
マニフェストへ反映する構成（トレイトのシグネチャは不変）。NodalDisp/MemberForce/Modal/
TimeHistory の書き込み・フィルタ付き照会・再オープン復元をテストで担保。

### 9.2 全 JobKind の実処理と `result_get`（squid-n-mcp）
- `analysis_run` が全 JobKind を実行: LinearStatic（NodalDisp+MemberForce を永続化、
  case=荷重ケース ID）、Eigen（Modal を case=0 で永続化。ResultKind で名前空間が
  分かれるため静的結果と衝突しない設計）、Pushover/TimeHistory（対応 ResultKind が
  無いためサマリのみ）、DesignCheck（MemberForce 永続化+NG 数サマリ）。
  引数は kind ごとの任意パラメータ（load_case/n_modes/dir/steps/…、既定は GUI と同一）。
- 永続化とジョブ Done への遷移を同一ロック内で行い、「Done なのに manifest に無い」
  状態を作らない。
- `result_get` ツール: manifest 存在確認→query→JSON 行配列（10,000 行で打ち切り+
  truncated フラグ）。node/member フィルタ・step_range 対応。
- `ServerState.results` は具象 `FsResultStore` 保持に変更（sync() を呼ぶため。
