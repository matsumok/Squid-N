# 検定比の3Dモデル上での可視化 申し送り

作成日: 2026-07-23
対象コード: `crates/squid-n-app/src/viewer/check_ratio.rs`（新規）、
`viewer/mod.rs`、`docs/result_view.md`

## 1. 何を実装したか

3D ビューアの表示モードに「検定比」を追加した（`ViewMode::CheckRatio`）。
`app.results.checks`（部材検定）・`app.results.joint_checks`（節点単位の検定：
柱梁接合部・パネルゾーン・耐震壁など）を集計し、部材・節点を検定比に応じて
着色する。詳細な表示仕様は利用者向けドキュメント `docs/result_view.md`
「検定比の表示」節を参照。

## 2. 設計判断

### 2.1 色規約は検定表（`design_view.rs`）と共通の `theme::status_color` を採用

検定表の検定比セルは既に `theme::status_color(ratio)`（≤0.8 緑／≤1.0 黄／
>1.0 赤=NG の3色）で着色されている。3D 可視化でも同じ関数をそのまま使い、
検定表と3Dビューで「同じ検定比なら同じ色」になるよう一貫させた。

**コンター（連続値グラデーション）は採らなかった**。N/Q/M 図のコンター表示
（`diagram.rs` の `contour_color`／`theme::ColorMap`）は値の相対的な大小を
連続色で見せることに意味があるが、検定比は 0.8・1.0 という**絶対的な判定
基準**を持つ値であり、「OK/注意/NG」の3値分類を一目で読み取れることの方が
重要と判断した。連続色にすると 0.79 と 0.81 が僅かな色差になり、判定基準
をまたいだことが視認しにくくなる。

### 2.2 集計関数を `egui::Painter` から独立させた

`max_ratio_by_elem` / `max_ratio_by_node`（`check_ratio.rs` 内、非公開）は
`(id, ratio, ok)` の軽量なタプル列を受け取り `HashMap<Id, (最大検定比, 全位置OKか)>`
を返す純粋関数として切り出した。`CheckResult`（`basis`/`detail` の
`String` を持つ）をそのまま受けると集計ロジックのテストに不要な文字列構築が
必要になるため、呼び出し側（`draw_check_ratio`）で `cr.ratio` / `cr.ok` へ
分解してから渡す設計にした。`diagram.rs` の `diagram_fill_polygons` /
`contour_color` を `egui::Painter` から切り離してユニットテストしている
既存の方針に倣っている。

内部実装は部材用・節点用で完全に同じ集計ロジック（最大値の採用・NG の
伝播）のため、`max_ratio_by_key<K, I>` というキー型ジェネリックな共通実装
1つに集約し、`max_ratio_by_elem` / `max_ratio_by_node` は型を固定した薄い
ラッパーにしている。

### 2.3 NodeId・ElemId のインデックス解決

部材の両端座標は `elem.nodes[i].index()` を直接 `pts` の添字として使う
（`ElemId`/`NodeId` の内部値が生成順にそのまま採番される前提。既存の
部材線描画・N/Q/M図が同じ前提で実装されている）。

一方、節点単位の検定（`joint_checks`）の対象節点を `pts` へ写像する際は、
念のため `app.model.nodes` を `enumerate()` して実際の配列位置を求め、
その位置の `node.id` が検定対象の `NodeId` と一致するかで突き合わせている
（`NodeId` の内部値が常に配列添字と一致するとは限らない前提に立った、より
安全な実装）。現状のモデル生成では両者は一致するはずだが、将来モデルの
節点配列が並び替え・フィルタされるようになっても壊れない書き方にした。

### 2.4 描画順序

部材線・壁の基本描画（灰色の部材線・青の壁）の後、選択ハイライト・支持
条件シンボルの前に検定比の着色を描く。検定比の色が部材線の上に重なり、
かつ選択中の部材（赤の太線）・支持条件シンボル（矢印・円弧）は検定比の色の
さらに上に重なって視認できるようにするため。

## 3. 残課題・注意

- 検定比図は「検定比」表示モード専用で、他モード（変形図・N/Q/M 図等）とは
  同時に重ねて表示できない（`viewer_panel` の `if mode == ViewMode::CheckRatio`
  分岐）。応力図に検定比を重畳する要望が出た場合は、現状の排他的なモード
  切替から「オーバーレイ」方式への変更を検討する必要がある。
- 節点検定の凡例集計（`max=`／`NG N件`）は部材検定・節点検定の両方を
  合算した値。`ViewMode::CheckRatio` の doc コメントは「部材検定の最大検定比
  で着色」とだけ書いているが、実装は節点検定（接合部等）の丸印も同じ凡例に
  含めている。将来、部材と節点で凡例を分けたいという要望が出た場合は
  `draw_legend` の集計元を分離する。
- 検定なしの部材・節点は色を一切変更しない（既存のグレー/青のまま）ため、
  「検定を実施していない」ことと「検定して OK だった」ことは色だけでは
  区別できない設計にしている（前者は無着色、後者は緑）。凡例に「未検定:
  グレー」の注記を入れることで対応した。

## 4. フェーズ A: CheckResult への式別内訳（components）の追加

対象コード: `crates/squid-n-design-jp/src/lib.rs`（`CheckKind`・
`CheckComponent`・`CheckResult::components` の追加）、および `CheckResult` を
生成する全箇所（`rc/`・`steel/`・`srrc/`・`cft/`・`brb.rs`・`joint_wiring/`
以下、約25箇所）。

### 4.1 何を実装したか

3D ビュー・検定表で「どの検定式（曲げ・せん断・付着等）が支配的か／NG か」
を後続フェーズで表示できるよう、`CheckResult` に式別の検定比内訳
`components: Vec<CheckComponent>` を追加した。`CheckComponent` は
`{ kind: CheckKind, ratio: f64 }` の組で、`CheckKind` は
`Bending`／`Shear`／`Bond`／`AxialBending`／`Axial`／`Deflection` の6種。

既存の `ratio`（`= max(各式の検定比)` に潰した値）・`ok`・`basis`・`detail`
は一切変更していない（後方互換）。`components` はあくまで表示用の付加情報。

### 4.2 設計判断: CheckKind の分類方針

- **既存の ratio 算定ロジックには手を入れず**、各検定関数内で「ratio の
  `max(...)` に入る前の値」を洗い出し、そのままの値を `components` に
  詰める方針にした（新たな計算式は導入しない）。
- **柱系（RC柱・SRC柱・CFT柱）の「軸力＋二軸曲げ」検定**は、軸力単独比
  （`ratio_axial`）と二軸曲げ複合比（`ratio_moment`/`ratio_biaxial`）を
  1つの `AxialBending` 成分（`= ratio_axial.max(ratio_moment)`）にまとめた。
  柱の軸力検定は曲げ検定と独立した式ではなく同じ N-M 相関曲線由来の
  同種の検定（軸力単独は M=0 の特殊ケース）であるため、表示上も1本の
  「軸+曲げ」として扱う方が利用者に分かりやすいと判断した。
  同様に二方向せん断（y/z 方向）も `Shear` 成分1つ
  （`= ratio_qy.max(ratio_qz)`）にまとめている。
- **鋼構造梁（steel/beam.rs）**は「組合せ検定（軸力+二軸曲げ）」「単独曲げ
  検定（My/Mz）」「von Mises 型合成応力度検定」の3種の曲げ系検定式が存在する
  が、`CheckKind` に梁専用の「複合曲げ」カテゴリを追加するほどの区別価値は
  ないと判断し、いずれも `Bending` 1成分（3者の `max`）にまとめた。
  せん断（強軸/弱軸）は `Shear` 1成分（両者の `max`）。
  von Mises 検定は軸力・曲げ・せん断応力を合成した式だが、H形の場合の式
  （`case_y`: σax+σb′とτyの合成、`case_z`: σax+σby+σbzとτzの合成）は
  曲げ応力度が支配的な項として現れるため `Bending` 側に含めている
  （曲げ系・せん断系のどちらに倒すか迷った箇所）。
- **冷間成形角形鋼管の柱梁耐力比検定**（`steel/cold_formed.rs`）は
  「柱の軸力低減耐力 ν・Zp・F」対「梁の全塑性モーメント和（+パネル耐力）」
  の比較であり、新しい `CheckKind` variant を増やすほどではないと判断し、
  柱の軸力＋曲げ耐力比という性質から `AxialBending` に分類した
  （判断に迷った箇所）。
- **BRB（`brb.rs`）**は軸力検定・座屈長さ検定の2式があるが、いずれも
  ブレース軸材の負担能力に関する検定のため `Axial` 1成分（両者の `max`）
  にまとめた。
- **接合部・耐震壁・水平接合面・パネルゾーンの単一せん断式**
  （`rc/joint.rs`・`rc/wall.rs`・`rc/horizontal_joint.rs`・
  `steel/panel_zone.rs`・`srrc/panel_zone.rs`・
  `joint_wiring/rc_joint.rs`・`joint_wiring/wall.rs`）は、式が1本しかない
  ため `components` は `Shear` 1成分（`ratio` そのもの）とした。
- **退化ケース**（Fc未設定・配筋情報なし・断面形状不一致等、`ok=true` の
  スキップ扱い）は `components: Vec::new()`（内訳情報なし）とした。

### 4.3 ratio との整合規則

「`components` が非空なら `components` の検定比の最大値が `ratio` と
（浮動小数の完全一致で）一致する」という不変条件を設けた。既存の `ratio`
算定が各式の `.max(...)` チェーンで構成されているため、そのチェーンに現れる
値をそのまま（式別に）`components` に詰め、複数の式を1つの `CheckComponent`
にまとめる場合も `.max(...)` の結合律により全体の `max` は変わらないことを
利用している（新たな計算誤差は生じない）。各モジュールのテストで
`components.iter().map(|c| c.ratio).fold(0.0, f64::max) == ratio` を確認
している。

## 5. フェーズ B: 検定比図の式別フィルタ・位置別マーカー・ホバー詳細

対象コード: `crates/squid-n-app/src/viewer/mod.rs`（`CheckRatioFilter` の
追加、ツールバーの検定式フィルタ行、ホバーによるツールチップ表示）、
`crates/squid-n-app/src/viewer/check_ratio.rs`（フィルタ適用・マーカー・
ツールチップ表データ生成のロジック）、`crates/squid-n-app/src/app/mod.rs`
（`check_ratio_filter`／`check_ratio_markers` の追加）、
`docs/result_view.md`。

### 5.1 何を実装したか

フェーズ A で `CheckResult` に追加された式別内訳（`components`）を、
3D ビューで実際に使えるようにした。

- **B-1 検定式フィルタ**: ツールバーに「検定式: 最大｜曲げ｜せん断｜…」の
  切替行を追加（`ViewMode::CheckRatio` のときのみ表示）。`app.check_ratio_filter`
  （既定 `Max`）で保持する。
- **B-2 位置別マーカー**: 部材の各検定位置に一辺 7px の正方形マーカーを重ね、
  NG の位置のみ数値を添える。`app.check_ratio_markers`（既定 true）で
  ON/OFF を切り替える。
- **B-3 ホバー詳細**: 部材にポインタを重ねると、検定位置×検定式の内訳表を
  ツールチップで表示する。
- **B-4 支配式ラベル**: フィルタ=最大のときのみ、部材中点のラベルに
  「1.13 せん断」のように支配式（部材内最大値を与えた位置の最大内訳）を
  併記する。

### 5.2 設計判断

#### 5.2.1 フィルタ適用を1つの純粋関数（`ratio_for_filter`）に集約した

「最大」と「式別」で表示する検定比・OK 判定のロジックが分岐するのは
着色・マーカー・ラベルの3箇所すべてに共通するため、`ratio_for_filter(cr,
filter) -> Option<(f64, bool)>` という1つの純粋関数に集約し、呼び出し側
（着色の集計・マーカー描画）はすべてこの関数を通す設計にした。式別フィルタ
で該当式が無い検定位置は `None` を返し、「フィルタ対象外＝未検定と同じ
無着色」という規則を一箇所（この関数の戻り値の意味）に閉じ込めている。

`Kind(k)` の OK 判定は `cr.ok`（その位置の全式を通した総合判定）ではなく
`r <= 1.0`（その式単独の判定）を使う。フィルタで特定の式だけを見ている
以上、判定もその式だけの基準で行うべきと考えたため（他の式が NG でも
選んだ式が OK なら、その式のフィルタ表示は OK にすべき）。

#### 5.2.2 集計関数を「(検定比, OK) の Option を返す位置列」ベースに一般化

フェーズA以前の `max_ratio_by_key` は `(id, ratio, ok)` の3要素タプル列を
受け取っていたが、フィルタ対象外（None）の位置を「その位置を集計に含め
ない」という形で表現する必要が生じたため、`(id, Option<(f64, bool)>)` を
受け取る形へ一般化した。全位置が `None` の部材・節点はマップに一切
エントリが作られず、結果として「未検定」と全く同じ扱い（無着色）になる。
既存のテストも `Some(...)` でラップする形に書き換え、加えて「全位置が
None なら未検定扱い」を確認するテスト（`max_ratio_by_elem_none_is_excluded`）
を追加した。

#### 5.2.3 検定式フィルタの選択肢を「結果に現れる式だけ」に絞った

`CheckKind` は6種あるが、部材種別（RC/S/SRC/CFT、梁/柱/ブレース）や検定
種別によって実際に現れる式は限られる（ブレースのみのモデルに「たわみ」
「付着」は現れない、等）。無関係な選択肢が常に6つ並ぶとフィルタの意味が
薄れるため、`available_check_kinds` という純粋関数を切り出し、部材検定・
節点検定の全 `components` を走査して実際に出現した `CheckKind` だけを
`CheckKind` の定義順で返すようにした。ツールバーはこの関数の戻り値だけを
ボタンとして描く。

#### 5.2.4 位置別マーカーは NG のみ数値を出す

全ての検定位置（柱フェイスi端・中央・柱フェイスj端に加え、ハンチ端・
継手・指定位置が加わると1部材あたり5〜6箇所になりうる）に数値ラベルを
出すと、3D ビュー上で文字が重なり合って読めなくなる。マーカー自体の色
（3段階規約）で OK/注意/NG は判別できるため、**数値が必要なのは基準を
超えた NG の位置だけ**と判断し、NG 以外はマーカーの色のみで表現すること
にした。

#### 5.2.5 マーカー位置はスクリーン座標の線形補間

検定位置 xi ∈ [0,1] のワールド座標を厳密に求めるには部材の3D座標を補間
してから再度投影する必要があるが、本ビューアの投影は正射影＋アフィン変換
（`project` 関数、回転・平行移動・一様スケールのみで透視変換を含まない）
のため、両端のスクリーン座標を xi で線形補間した点は、ワールド座標を
xi で補間してから投影した点と完全に一致する。既に投影済みの両端点
（`pts` 由来の `p0`/`p1`）がある場面では、この性質を使って再投影を省略し、
スクリーン座標の補間だけで正しい位置を求めている。

#### 5.2.6 ホバー詳細のヒット判定は既存のクリック選択と同じロジックを再利用

「通常モードのクリックで部材を選ぶ」ロジック（`dist_point_to_segment` に
よる最近傍部材探索、8px 閾値）と全く同じ考え方をホバー判定にも使うことで、
「検定比図でも普段のクリック選択と同じ部材が選ばれる」という一貫した操作
感にした。ViewCube 上のホバー中はツールチップを出さない（`cube_hover`
が `Some` の間は判定自体をスキップ）ようにし、ViewCube のクリック判定と
同じ除外条件を踏襲している。

#### 5.2.7 ツールチップの表データ生成を純粋関数化した

`egui::show_tooltip_at_pointer` はウィジェットに紐付かない任意位置への
ツールチップ表示という用途に対して他に簡潔な代替が無いため、egui 0.34 で
非推奨（`Tooltip` 型への移行が推奨）であることを承知の上で使用し、
`app/panels.rs` の既存箇所と同じ方針で `#[allow(deprecated)]` を付けている。

一方、「位置×式の表データをどう組み立てるか」というロジック自体は
`egui::Painter`／`egui::Ui` に一切依存しない形（`build_tooltip_rows`）に
切り出し、`Vec<(xi, ok, components)>` を受け取って
`(列に出す CheckKind の集合, 各行の値・判定)` を返す純粋関数としてテスト
している。`elem_check_positions`（部材IDで `results.checks` を絞り込んで
軽量なタプル列に変換する関数）と組み合わせることで、`CheckResult` を
描画コードに渡す前に軽量な形へ変換する既存の設計方針（フェーズA以前の
「`egui::Painter` から独立させた集計関数」と同じ考え方）を踏襲している。

### 5.3 残課題・注意

- ホバー詳細は部材（線材）のみを対象とし、節点単位の検定（接合部・
  パネルゾーン・耐震壁）にはツールチップを出していない。柱梁接合部等の
  詳細確認が必要になった場合は、節点円のヒット判定を別途追加する必要が
  ある。
- 位置別マーカーの座標補間はスクリーン座標の線形補間（5.2.5）に依存して
  おり、将来ビューアに透視投影を導入した場合はこの前提が崩れる。導入時は
  マーカー描画をワールド座標補間＋再投影の方式に直す必要がある。
- `ratio_for_filter` の `Kind(k)` 判定基準（`r <= 1.0`）は検定表
  （`design_view.rs`）の内訳列の着色（`theme::status_color`。0.8/1.0の
  3段階）とは判定粒度が異なる（内訳列の色は3段階、フィルタのOK/NGは
  2値）。これは3Dビューの着色が「OK/注意/NGの3段階」で統一されている
  一方、フィルタの「対象位置か否か・NGかどうか」の判定は2値で十分という
  設計判断だが、将来「フィルタ適用時も0.8基準の注意色を出したい」という
  要望が出た場合は、`ratio_for_filter` の戻り値を3値（enum）に拡張する
  検討が必要になる。

## 6. フェーズ C: 検定表（`design_view.rs`）の内訳列

対象コード: `crates/squid-n-app/src/design_view.rs`。

部材検定表（「部材検定（許容応力度）」）に「内訳」列を追加し、`cr.components`
を「曲げ 0.82／せん断 1.13／付着 0.44」のように式ごとに横並び・色付きで
表示するようにした。内訳の色分けは検定比図・検定表の検定比セルと同じ
`theme::status_color` を使い、3D ビューと表とで式別の状態表示を一貫させて
いる。抽出タプル（`checks`）に `cr.components.clone()` を追加しただけの
小さな変更のため、専用の純粋関数は切り出していない（内訳セルの組み立ては
`ui.horizontal` 内の単純な繰り返し表示のみ）。

節点単位の検定表（`joint_checks`）はほぼ単一式（せん断1本等）のため、
内訳列は追加していない。

## 7. フェーズ D: 後方互換制約の撤廃に伴うリファクタリング

対象コード: `crates/squid-n-design-jp/src/lib.rs`（`CheckResult`/`CheckOutcome`）、
`CheckResult` を生成する全箇所（rc/steel/srrc/cft/brb/joint_wiring 以下、
約25箇所）、`crates/squid-n-app/src/app/mod.rs`（`ResultsBundle`・
`PositionCheck`/`MemberChecks`/`JointCheck`・`group_member_checks`）、
`app/actions.rs`（`run_design_check`）、`viewer/check_ratio.rs`、
`design_view.rs`、`summary.rs`、`viewer/mod.rs`、`app/panels.rs`、
`app/tests.rs`、`squid-n-mcp/src/job/design_check.rs`。

ユーザーから「後方互換は不要」との明確な指示を受け、フェーズA〜Cで
`components` を追加した際に後方互換維持のため抱えていた3つの歪みを
解消した。

### 7.1 単一情報源化: `ratio`/`ok` を導出メソッドへ

フェーズAで `CheckResult` に `components` を追加した際、既存の
`ratio`/`ok` フィールドは後方互換のため残し、「`components` が非空なら
その最大値が `ratio` と一致する」という不変条件を約25箇所の検定関数が
手作業（`.max(...)` チェーンの結果を両方に書く）で維持していた。この
二重管理をやめ、`ratio`/`ok` フィールドを削除して次のメソッドに置き換えた:

```rust
pub struct CheckResult {
    pub basis: String,
    pub detail: String,
    pub components: Vec<CheckComponent>,  // 1件以上
}
impl CheckResult {
    pub fn ratio(&self) -> f64 { /* components の max */ }
    pub fn ok(&self) -> bool { self.ratio() <= 1.0 }
}
```

不変条件は「`ratio()` の定義そのもの」になったため型で保証され、各検定
関数側で維持する責務が消えた。フェーズAで追加した「`components` の
max と `ratio` の一致」を確認するテスト（`test_*_max_matches_ratio` 系）は、
確認対象の不変条件が型で保証されるようになったため、内訳の内容
（どの `CheckKind` が入るか）だけを検証する形に簡略化した（テスト名も
`_max_matches_ratio` サフィックスを外した）。

副次的な確認として、`ok` の判定ロジックが `ratio <= 1.0` 以外だった箇所
（`cft/mod.rs`・`srrc/beam.rs`・`srrc/column.rs` の `ratio <= 1.0 &&
ratio.is_finite()`）が4箇所あったが、`ratio` は常に非負の実数（無限大には
`ratio_or_large` が 1e9 という有限の代用値を使うため到達しない）であり、
`ratio` が `NaN`/`+inf` なら `ratio <= 1.0` は比較演算の定義上すでに
`false` になる。したがって `&& ratio.is_finite()` は恒等的に冗長であり、
`ok()` を単純に `ratio() <= 1.0` とすることに伴う挙動変化は無い。

### 7.2 検定不能を型で明示: `CheckOutcome`

「Fc 未設定」「配筋情報なし」「断面形状不一致」の退化ケース（`rc/mod.rs`・
`cft/mod.rs`・`srrc/mod.rs` の `RcDesign`/`CftDesign`/`SrcDesign::check` に
計6箇所）は、フェーズA以前から `components: Vec::new()`・`ratio: 0.0`・
`ok: true` という「検定した体で検定比0・OKを返す」実装だった。3Dビュー・
検定表はこれを「検定して問題なし（緑）」と区別できず、入力不足を見逃す
リスクがあった。

これを型で解消するため、検定結果を次の列挙型で表すようにした:

```rust
pub enum CheckOutcome {
    Checked(CheckResult),
    Skipped { reason: String },
}
```

`DesignCheck::check`（`RcDesign`/`CftDesign`/`SrcDesign`/`SteelDesign` の
4実装）の戻り値を `CheckResult` から `CheckOutcome` に変更し、退化ケースは
`Skipped { reason }` を返すようにした（`reason` には従来 `basis` に
入れていた「RC 検定: Fc 未設定」等の文言をそのまま流用した）。これにより
`CheckOutcome::Checked` に包まれた `CheckResult` は **`components` が
必ず1件以上**という前提を型のドキュメントコメントとして明記できるように
なった（実際に空の `components` を返す経路は残っていない）。

BRB 検定（`brb::brb_check`）・節点単位の検定配線（`joint_wiring::
collect_joint_checks*`・`rc::horizontal_joint::collect_pca_checks`）は
元々「対象外なら該当エントリ自体を push しない」実装のため退化ケースを
持たず、呼び出し側（`app/actions.rs`・`squid-n-mcp`）で
`CheckOutcome::Checked(cr)` に包むだけで足りる。

表示側の変更（本リファクタで唯一の外部仕様変更）は `docs/result_view.md`
「検定不能の表示」節を参照。要点は「未検定と同様に無着色・NG件数に
含めない・理由を表示する」。

### 7.3 `ResultsBundle` の検定結果をタプルからグループ化構造へ

`ResultsBundle.checks: Vec<(ElemId, f64, CheckResult)>` は検定位置ごとの
フラットな列で、部材単位の情報（ホバー詳細・検定結果数表示等）を得る
たびに `results.checks.iter().filter(|(id,_,_)| *id == elem_id)` という
全件走査が各所（`check_ratio.rs` の `elem_check_positions`・`show_check_
tooltip`、`app/panels.rs` のインスペクタ）に散らばっていた。

これを部材単位のグループ化構造に変更した:

```rust
pub struct PositionCheck { pub xi: f64, pub outcome: CheckOutcome }
pub struct MemberChecks { pub elem: ElemId, pub positions: Vec<PositionCheck> }
pub struct JointCheck { pub node: NodeId, pub label: String, pub outcome: CheckOutcome }

pub member_checks: Vec<MemberChecks>,  // ResultsBundle
pub joint_checks: Vec<JointCheck>,
```

配置は squid-n-app 側（`app/mod.rs`、`ResultsBundle` の近く）にした。
squid-n-mcp（`design_check.rs`）は検定結果を集計値（`n_checks`/`n_ng`/
`max_ratio` 等）に畳み込むだけで、部材単位のグループ構造自体は使わない
（`checker.check(...)` の戻り値 `CheckOutcome` を直接 `match` して集計する）
ため、共有型として squid-n-design-jp 側へ引き上げる必要はないと判断した。

生成側（`app/actions.rs` の `run_design_check`）は、検定位置ごとに
`(ElemId, f64, CheckOutcome)` のフラット列を組み立てたあと（BRB 検定・PCa
水平接合面検定の結果もこの時点で `CheckOutcome::Checked` に包んで合流
させる）、新設のヘルパー `group_member_checks` で部材単位にグループ化
しつつ各部材内を `xi` 昇順にソートする。部材の出現順は入力列の初出順を
保つ。

利用側の簡素化:

- `check_ratio.rs`: `checks_by_elem` は `HashMap<ElemId, Vec<(f64, &CheckResult)>>`
  を毎回組み立てる実装から `HashMap<ElemId, &MemberChecks>`（部材IDから
  グループ化済みの `positions` を直接参照するだけ）に変わり、
  `elem_check_positions` も同様に「該当部材のグループを探して
  `positions` をそのまま返す」だけの実装になった（ホバーごとの全行
  線形走査が不要になった）。
- `design_view.rs`／`summary.rs`／`viewer/mod.rs`／`app/panels.rs`／
  `squid-n-mcp` は、`member_checks`/`joint_checks` を辿って
  `CheckOutcome::Checked(cr)` なら `cr.ratio()`/`cr.ok()` を、
  `CheckOutcome::Skipped { reason }` なら「検定不能」表示に切り替える
  形にそれぞれ書き換えた。

## 8. フェーズ E: detail の式別分割

対象コード: `crates/squid-n-design-jp/src/lib.rs`（`CheckComponent::detail`
の追加、`CheckResult::detail` のドキュメント更新）、`CheckResult` を生成
する全箇所（rc/steel/srrc/cft/brb/joint_wiring 以下、フェーズD同様の
約20箇所）、`crates/squid-n-app/src/design_view.rs`（検定表のホバー表示）、
`docs/result_view.md`。

### 8.1 何を実装したか

`CheckResult::detail` は従来、曲げ・せん断・付着など複数の検定式の数値
根拠（許容値・作用値・中間係数）が1本の長い文字列に連結されていた。これを
式別に分割し、`CheckComponent` に `detail: String` を追加して式固有の
根拠を持たせた（`CheckResult::detail` は「全検定式に共通の数値根拠
（断面諸元など）」のみを持つよう役割を狭めた）。`detail` フィールドが
増えたことで `CheckComponent`/`CheckResult` の `Copy` 実装は外れ、`Clone`
のみになった（利用側は `.clone()` で追従）。

検定表（`design_view.rs`）の内訳セル（各式のラベル）に
`.on_hover_text(&c.detail)` を、根拠セル（basis）に共通 detail が
非空の場合のみ `.on_hover_text(&r.detail)` を付け、節点単位の検定表
（`joint_checks`）も根拠セルへ（共通 detail ＋ 単一 component の detail
の連結）をホバー表示するようにした。3Dビューのホバーツールチップ
（`check_ratio.rs`）は既存のコンパクトな表のまま変更していない
（詳細な数値根拠は検定表側で確認する役割分担）。

### 8.2 断片の分類方針

各検定関数の既存 `detail` 文字列を構成する断片（`"MA_t={:.1} N·mm"` の
ような `format!` の一片）を、次の基準で「共通」「式別（Bending/Shear/
Bond/AxialBending/Axial）」に仕分けた:

- **その断片が、どの式の検定比（`ratio_*`）の算定に使われている値か**を
  最優先の基準にした。例えば RC 梁の `pw`（せん断補強筋比）はせん断耐力
  `QA` の算定にのみ使われるため Shear、`MA_t`/`MA_c`/`MA` は曲げ耐力
  算定にのみ使われるため Bending、というように、コード上の使用箇所を
  たどって機械的に判定した。
- **複数の式で共有される断面諸元**（RC 梁の `at`・`d`・`j`、RC 円形柱の
  `at`・`d` 等）は共通 detail に置いた。これらは曲げ・せん断・付着の
  いずれの式でも参照する値であり、どちらか一方の component に押し込む
  と「もう一方の式を見るときにこの値が見えない」という非対称が生じる
  ため。矩形柱（`rc/column/mod.rs`）は強軸・弱軸で断面諸元
  （`axis_z.props`/`axis_y.props`）が別々のため、共有される断面諸元が
  無く共通 detail は空文字列にした。
- **検定比の算定に使われない参考情報**（鋼構造梁の必要横補剛数・たわみ）
  は、どの式にも属さないため共通 detail に置いた。
- **単一式の検定**（wall・joint・panel_zone・brb・cold_formed・
  horizontal_joint・joint_wiring 配下等）は、分割の余地が無い（component
  が1件のみ）ため全文をその component の detail に移し、共通 detail は
  空文字列にした。

### 8.3 判断に迷った断片とその根拠

- **RC 梁の bond_detail（付着検定の詳細）**: `bond` が `None`（`ctx.length`
  未設定で付着検定を省略）の場合、従来は「付着検定: 部材長(柱面間距離)が
  未設定のため省略」という文言も detail の一部だった。この場合 Bond
  component 自体が存在しない（`components` に push されない）ため、
  省略の旨を置く先が無い。`Bond` component を「検定していないのに存在
  する」形で追加すると `components` の意味（=実施した検定式の内訳）が
  崩れるため、これは例外的に共通 detail の末尾に置くことにした
  （課題本文の指示どおり）。
- **鋼構造梁・鋼構造柱の von Mises 型合成応力度検定**: 合成応力度は
  軸力応力度 σax・曲げ応力度 σby/σbz・せん断応力度 τ を全て使うため、
  Bending/Shear のどちらに属するか自明ではない。コード上
  `ratio_bending = ratio_comb.max(ratio_my).max(ratio_mz).max(mises_ratio)`
  （steel/beam.rs）／`ratio_axial_bend` に mises 相当の式は含まれず
  `ratio_shear` 側に von Mises 型の式が入る（steel/column.rs）というように
  **既存コードが `Bending`/`Shear` のどちらの ratio に mises 系の値を
  合流させているか**で判定した（新たな分類基準を持ち込まず、既存の
  ratio 集約ロジックにそのまま従う）。その結果、σax・σby・σbz・fc・fb
  はいずれも Bending 側（steel/beam.rs は mises も Bending の ratio に
  含むため）、τ・fs は Shear 側に分類した。
- **CFT柱・SRC柱の「s_pc・fc'」（鋼管フランジ食い込みによる fc' 低減）**:
  RC部分の圧縮耐力 rNc（AxialBending側）の算定にのみ使う中間値であり、
  せん断側の計算には登場しないため AxialBending に分類した。
- **SRC梁・SRC柱の「設計用せん断力の決定方式」注記（構造規定方式／弾性
  分担）**: せん断力そのものの値ではなく「どちらの方式で Qd を決めたか」
  という注記だが、せん断検定の前提条件を示す情報のため Shear に分類した。

### 8.4 同一性の検証（`full_detail`）

分割の前後で情報が失われていないことを機械的に保証するため、
`squid-n-design-jp` の `#[cfg(test)]` ヘルパーとして `lib.rs` に
次の関数を追加した:

```rust
/// 共通 detail と全式の detail を連結する（分割で情報が失われていないことの検証用）。
#[cfg(test)]
pub(crate) fn full_detail(cr: &CheckResult) -> String {
    let mut s = cr.detail.clone();
    for c in &cr.components {
        s.push_str(", ");
        s.push_str(&c.detail);
    }
    s
}
```

分割前から存在した `result.detail.contains(...)` という断言はすべて
`crate::full_detail(&result).contains(...)` に置き換え、1件も削除せず
全件維持した（対象: `rc/beam.rs`・`steel/column.rs`・`steel/beam.rs`・
`srrc/column.rs`・`joint_wiring/tests.rs`・`cft/tests.rs`。
`full_detail(&X).contains(...)` の形の断言は合計23件）。
これにより「分割前に detail に含まれていた文字列が、分割後もどこか
（共通 detail か、いずれかの component の detail）に必ず存在する」ことが
テストで保証される。

加えて、代表的な検定関数（rc/beam・steel/beam・steel/column・cft/box）
には「断片が意図した component に配置されている」ことを直接確認する
テストを追加した（例:
`test_beam_check_detail_fragments_assigned_to_intended_components` は
Bending の detail に `"MA_t="` が含まれ Shear の detail には含まれない
こと、およびその逆（`"QA="`）を確認する）。
